MuduDB 是一个数据库系统，帮助你更轻松地开发面向数据的应用，并将应用直接部署和运行在数据库之上。

**这个项目目前处于早期开发阶段，仅作演示用。**

它引入了一系列[创新特性](doc/cn/innovative.cn.md)，利用现代 AI 与云技术来提升数据系统开发效率，并优化资源利用率。

## 架构

<img src="doc/pic/architecture.svg" width="100%">

上图展示了 MuduDB 的整体架构。

MuduDB 采用内核-运行时架构，将传统上应用逻辑与数据管理之间的边界收敛到一个统一的执行环境中。

内核提供最小但完整的正确性基础设施，包括存储、事务处理和查询执行。
它并不暴露传统 API 连接驱动那样的接口面，而是定义了一组精简的[系统调用](doc/cn/syscall.cn.md)接口。
这些接口用于完成数据访问和会话管理。

运行时层通过 [WebAssembly Component Model](https://component-model.bytecodealliance.org/) 承载用户定义的过程（WebAssembly）。
运行时被刻意设计为不负责调度：
它不会引入独立的并发机制或执行策略，
而是作为一个被系统调用的被动执行引擎。
这样可以避免在不同层重复实现调度逻辑，并将控制权集中保留在内核中。

在用户层面，MuduDB 只支持顺序且同步的编程模型（不支持并发和异步）。
开发者使用通用编程语言编写面向过程的代码（①），
这不同于传统存储过程，它可以通过客户端连接以交互方式直接调用（②）。
MuduDB 的工具链能够将这类过程转换为可部署的产物：
同步代码会被转换为异步形式（③），随后编译为 WebAssembly，并与相关资源（例如 schema 定义和初始数据）一起打包。

在运行时，过程调用（④）会在内核的工作线程中触发异步执行（⑤）。
执行期间发出的系统调用（由 WASM host 实现的接口函数）会陷入内核，并在事务与调度控制之下执行（⑥）。
因此，计算与数据访问被放置在同一位置，消除了关键路径上的跨边界交互。

执行围绕每核一个 worker 的模型组织，每个 CPU 核心对应一个专用工作线程。
所有 I/O、网络处理以及用户代码执行都通过协作式调度复用在这些 worker 中。
这种设计避免了线程间同步、加锁以及抢占式上下文切换，从而降低协调开销并提升缓存局部性。
