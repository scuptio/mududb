# 面向现代硬件优化的架构

本文介绍一种面向现代硬件优化的数据库运行时架构。它的目标并不仅仅是把传统同步服务器改写成异步语法，而是围绕现代 Linux 内核能力、多核 CPU、批量 I/O、缓存局部性以及更低的锁竞争，重新组织整条数据路径。

在 MuduDB 的语境中，这种架构主要体现为：

- 使用 `io_uring` 作为主要 I/O 模型
- 使用由状态机和 continuation 驱动的异步请求生命周期
- 使用按核划分的 worker 与分区本地所有权，以降低跨核竞争
- 在热点路径上使用无锁或低锁结构
- 对提交、完成、唤醒和响应处理进行批量化

这是一篇架构概览文档。它重点介绍设计意图、系统结构以及性能上的理由。

---

## 1. 为什么需要面向现代硬件优化的架构

传统 OLTP 服务器设计通常具有以下特征：

- 网络 I/O 与执行调度分散在多层抽象之中
- 请求生命周期依赖线程切换、共享队列和锁竞争
- accept、read、parse、execute 和 write 被实现为松散连接的阶段
- 同一个连接或 session 可能在多个线程之间反复迁移

这些模型当然可以工作，但在现代多核机器上，它们往往会暴露出一些瓶颈：

- 更严重的锁竞争
- 更高的缓存未命中率
- 更差的尾延迟
- 过多的系统调用边界
- 关键路径中的调度器开销

面向现代硬件优化架构的核心前提是：

1. 现代机器已经具备了很强的核级并行能力。关键问题不再只是“有没有并发”，而是“并发是否保持局部性”。
2. Linux 现在已经提供了诸如 `io_uring` 这类更低开销的异步 I/O 能力，可以将网络、文件、超时和取消统一到一个完成模型之下。
3. 高吞吐的热点路径应该避免“大共享状态加重锁”的设计，而应该依赖显式所有权、本地数据以及最小化的跨核消息传递。

## 2. 核心设计原则

这类架构通常遵循以下原则：

- `I/O first`：I/O 模型是顶层设计约束，而不是事后才做的优化
- `ownership first`：每个连接、任务和状态对象都应有明确的所有权
- `locality first`：数据、队列和执行上下文应尽可能停留在同一个 CPU 核附近
- `batch first`：提交、完成、唤醒和刷新操作都应适合批量化
- `async by runtime`：异步复杂度应由编译器与运行时吸收，而不是直接压给应用作者

典型目标形态如下：

- 一个 bootstrap / control plane
- 多个 worker 线程
- 每个 worker 绑定到一个 CPU 核或一个 partition
- 每个 worker 拥有自己的 I/O ring、本地队列、连接表和执行器

## 3. `io_uring` 的角色

`io_uring` 是这套架构的基础。它不仅仅是“更快的异步 socket API”。它将 I/O 从“阻塞式系统调用加休眠线程”的模型，转变为统一的“提交队列 + 完成队列 + 用户态事件循环”模型。

### 3.1 两个核心队列

- `SQ`，即 Submission Queue
  - 用户态向内核描述并提交 I/O 请求
- `CQ`，即 Completion Queue
  - 内核将完成的事件回报给用户态

这使得 worker 可以在统一的事件流中处理多种操作：

- `accept`
- `recv`
- `send`
- `read`
- `write`
- `fsync`
- `timeout`
- `cancel`

### 3.2 为什么它适合数据库

数据库通常需要同时处理：

- 高频网络流量
- WAL 或日志写入
- 数据文件 I/O
- 超时、取消和背压

`io_uring` 的关键价值在于，它可以把这些操作统一到同一个由完成事件驱动的运行时模型中。系统因此可以围绕“事件完成”组织，而不是围绕“线程被唤醒”组织。

### 3.3 对性能的直接影响

如果实现得当，`io_uring` 通常能带来：

- 更少的系统调用切换
- 更自然的批量提交与批量完成
- 更少的阻塞和更少的上下文切换
- 更干净的 worker 本地事件循环
- 同时覆盖网络与文件 I/O 的统一运行时

## 4. 按核划分的 Worker 架构

面向现代硬件优化的运行时通常不是“一个通用线程池，任意线程窃取任意任务”，而更接近于“每个 CPU 核拥有一个明确的所有权域”。

### 4.1 Worker 视角

每个 worker 通常拥有：

- 一个 `io_uring`
- 一个 partition 本地连接表
- 一个本地 ready queue
- 一个跨 partition inbox
- 一个协议状态机执行器
- 一个过程或组件运行时上下文



### 4.2 为什么按核所有权很重要

按核设计的主要目的，是尽量减少共享写路径。

如果一个连接在其大部分生命周期里都停留在同一个 worker 上：

- 它的协议状态就不需要在线程之间迁移
- 它的接收和发送缓冲区就不会在不同 CPU 核之间频繁导致缓存失效
- 它的任务推进不依赖共享锁
- 它的本地队列和本地内存会更符合缓存局部性

这种模型比“任意线程都能处理任意连接”更受约束，但在负载下可预测性更高。

## 5. 异步处理模型

现代异步架构并不意味着“到处写 `async/await` 就结束了”。真正关键的是，请求生命周期必须能够被拆解、挂起、恢复，并且能由批量事件驱动。

### 5.1 异步的本质

异步处理首先不是语法问题，而意味着：

- 一个请求在等待 I/O 时不会阻塞 worker
- 执行可以在之后从 continuation 或状态机恢复
- 前进推进由完成事件驱动

这对数据库过程或组件执行尤为重要，因为业务逻辑往往天然就是顺序式的：

1. 读取参数
2. 发起查询或 KV 操作
3. 等待结果
4. 更新状态
5. 返回响应

如果要求开发者把这一切都手工编码成显式的底层状态机，复杂度会大幅上升。更好的模型是：

- 开发者按接近顺序语义的方式编写逻辑
- 编译器和运行时把阻塞点转换为挂起点
- 当相应的 `io_uring` 完成事件到达时，worker 再恢复执行

### 5.2 面向用户的同步 API，内部异步运行时

一个重要的设计选择是：不应将异步接口直接暴露给用户。

从用户视角看：

- 过程是基于看起来像同步的接口来编写的
- 控制流保持顺序化，更容易理解和推理
- 业务逻辑不需要直接依赖 future、显式唤醒或手工异步状态处理

在部署时，转译器会把这种同步过程代码降级为适合运行时执行的异步过程代码。执行模型如下：

1. 用户编写同步风格的过程逻辑
2. 转译器把它改写为异步控制流
3. 部署后的过程由运行时以 continuation 驱动的异步方式执行

这种分层是有意为之：

- 用户保留简单的同步编程体验
- 运行时仍然能够在 `io_uring` 之上获得真正的异步执行能力

系统正是通过这种方式，在不要求每个过程作者直接编写底层异步代码的前提下，保留异步性能特征。

### 5.3 事件驱动的执行链

一个典型的执行链如下：

```text
recv CQE
 -> decode request
 -> build execution task
 -> invoke procedure / syscall
 -> if pending: park continuation
 -> submit next I/O
 -> later CQE arrives
 -> wake continuation
 -> finish result encode
 -> send response
```

它的优势在于，worker 主循环保持为一个“状态推进引擎”，而不是一个“阻塞等待引擎”。

同样的流程也可以从编程模型角度理解：

```text
user writes synchronous procedure
 -> transpiler converts it into async procedure form
 -> runtime executes it as continuation-driven control flow
 -> io_uring completions resume execution at suspension points
 -> response is encoded and sent
```

## 6. 无锁与低锁设计

“无锁”不应被理解为“系统中任何地方都绝对没有锁”，而应理解为：

- 在热点路径上尽可能不使用互斥锁
- 跨核通信主要通过单向消息队列完成
- 尽量减少共享可变状态
- 用显式所有权转移替代隐式共享访问

### 6.1 为什么热点路径必须避免锁

在高并发数据库系统中，真正昂贵的通常不是某一次单独的加锁操作，而是：

- 锁竞争带来的自旋或休眠
- 缓存行在不同 CPU 核之间来回抖动
- 热点共享结构让整体延迟受最慢线程拖累

因此，现代系统通常更偏好：

- worker 本地连接表
- worker 本地 ready queue
- 仅在必要时迁移连接
- 用无锁 MPSC inbox 处理远程 handoff

### 6.2 哪些结构最适合无锁设计

通常最适合采用本地无锁设计的结构包括：

- ready task queue
- continuation wake queue
- 跨 partition 的消息 inbox
- 连接 handoff 队列

而不必激进去锁的结构包括：

- 配置管理
- 启动与关闭流程
- 低频元数据更新

换句话说，工程投入应集中在真正的热点路径上。

### 6.3 无锁设计依赖所有权

如果没有所有权约束，无锁设计通常会退化为更难调试的共享状态并发问题。

正确的顺序应该是：

- 先定义 worker 所有权
- 再定义连接所有权
- 再定义 session 所有权
- 再定义任务所有权
- 然后才去优化队列、slab、arena 和 handoff 机制

## 7. 批处理、背压与流水线

面向现代硬件优化架构的另一个关键特性是批处理。

### 7.1 为什么批处理重要

`io_uring` 特别适合以下批量操作：

- 批量提交 SQE
- 批量收集 CQE
- 批量唤醒 continuation
- 批量写回响应

批处理可以减少：

- 系统调用频率
- 队列访问开销
- 调度抖动

### 7.2 背压必须成为设计的一部分

一个高性能系统不能只关注“如何更快接收更多工作”，也必须明确“在什么情况下延迟、拒绝或放慢工作”。

典型的背压点包括：

- ring 深度接近饱和
- 本地 ready queue 过长
- 慢请求类别持续堆积
- 当写吞吐跟不上时，发送缓冲持续累积

因此，背压应该作为运行时的一等公民来设计，而不是在系统出问题后再临时打补丁。

---

## 8. 协议面与当前 SQL 状态

对于当前的 `io_uring` 架构，应将运行时路径理解为“KV 优先”，而不是“SQL 优先”。

这意味着当前方向主要围绕以下内容展开：

- 更轻量的自定义协议处理
- partition 本地执行
- 以 `get`、`put` 和 `range` 为代表的 KV 操作


需要准确说明当前状态：

- `io_uring` 运行时架构已经存在
- 基于 continuation 的异步执行模型已经存在
- 面向 KV 的路径已经存在
- 当前 `io_uring` 路径上的 SQL 支持尚未实现

这一运行路径上的 SQL 支持仍在开发中。

因此，目前应理解为：

- 在 `io_uring` 后端中，SQL 仍是一项未完成能力
- `io_uring` 运行时当前可用的主要方向是面向 KV 的路径

---

## 9. 这套架构的主要收益

如果完整实现，这套架构应当在以下方面带来收益：

- 更高的吞吐量
  - 更短的 I/O、调度和执行路径
- 更低的尾延迟
  - 更少的锁竞争与调度噪声
- 更好的可扩展性
  - 随着核心数增加，更自然地按 partition 扩展
- 更好的缓存局部性
  - 连接、状态和任务尽可能保持在 worker 本地
- 更可预测的运行时行为
  - 所有权边界与 handoff 边界更加明确

---

## 10. 代价与工程权衡

这种架构并不是零成本的。它会引入真实的工程复杂度：

- 状态机与 continuation 设计会更困难
- 调试难度高于简单的阻塞模型
- 错误恢复路径必须极其清晰
- 连接迁移、取消、超时和关闭路径会更加微妙
- Linux `io_uring` 支持会成为一个重要的平台依赖

一个更务实的工程策略通常是：

1. 先定义运行时目标与所有权模型
2. 再构建 worker 主循环与协议推进逻辑
3. 逐步把更多 I/O 操作统一到同一个完成模型之下

---

## 11. 这对 MuduDB 意味着什么

对于 MuduDB 而言，面向现代硬件优化的架构价值并不仅仅在于“更快”。它还使系统能够同时保留以下三个重要特性：

- 对过程作者来说自然的同步编程体验
- 对内部而言真正异步且感知多核的运行时
- 向组件化、KV 优先、低延迟执行路径持续演进的空间


如果把这套设计总结成一句话，那就是：

> 以 `io_uring` 作为统一的异步完成模型，以按核 worker 作为局部性边界，以无锁队列降低共享竞争，并通过运行时和转译器而非用户代码来吸收异步复杂度。
