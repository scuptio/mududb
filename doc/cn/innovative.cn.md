# MuduDB 创新特性

MuduDB 重新思考了应用与数据库之间的边界。它不再把数据库当作被动的存储层，而是让开发者使用通用编程语言编写普通的过程，在数据库内核内部运行，并以可版本化的包进行部署。最终得到的系统既具备现代应用开发的开发体验，又拥有计算与数据共置带来的性能和正确性优势。

---

## 1. AI 辅助数据库工程

MuduDB 利用大语言模型加速数据库工程。它可以根据高层描述生成实体关系图、DDL 脚本和过程代码。由于生成的代码使用带类型的通用编程语言编写，因此能在进入服务器之前通过编译期校验，降低 AI 生成逻辑在运行时才暴露错误的风险。

**亮点**

- 根据自然语言需求生成 ER 图
- 生成 DDL 脚本和表定义
- 生成存储过程与辅助函数
- 在编译期验证 AI 生成的代码

## 2. Mudu Procedure（MP）

[Mudu Procedure](procedure.cn.md) 是 MuduDB 的核心编程模型。一个用 Rust 或 AssemblyScript 编写的函数，既可以在开发阶段以交互方式调用，也可以作为服务端过程部署到生产环境。转译器会把看起来同步的源码转换为 WebAssembly 运行时所需的异步包装代码，开发者无需手动管理异步状态机。

**亮点**

- 一套代码同时用于交互式开发与生产部署
- 过程在数据库内核内部贴近数据运行
- 开发者体验是同步的，运行时执行是异步的
- 同时支持临时事务和结构化工作流

## 3. 面向现代硬件优化的架构

MuduDB 面向现代多核服务器和 NVMe 存储设计。它以 Linux [io_uring](modern_hardware.cn.md) 作为统一的异步 I/O 模型，为每个 CPU 核心分配一个工作线程，并尽可能让连接、队列和状态都保持在该 worker 本地。用户过程以顺序风格编写，但会以由 io_uring 完成事件恢复的续体形式执行。

**亮点**

- 每核独立 worker，边界清晰
- 通过 `io_uring` 统一网络与文件 I/O
- 热路径采用无锁或低锁结构
- 基于续体的执行，不阻塞 worker 线程

## 4. 微内核架构设计

MuduDB 保持核心引擎小巧且专注于 essentials：存储、ACID 事务、查询解析和查询执行。其他能力都位于可独立演进的插件生态中。这种分离让内核更易于理解和测试，同时为 JSON/图支持、外部运行时模块、自定义存储引擎等扩展留出空间。

**亮点**

- 小巧专注的核心引擎
- 支持扩展与自定义运行时的插件生态
- 无需修改内核即可添加自定义存储引擎
- 更清晰的所有权边界和更小的可信计算基

## 5. 使用通用编程语言编写过程

MuduDB 过程是 Rust 或 AssemblyScript 中的普通函数，用 `/**mudu-proc**/` 指令标记即可。开发者可以继续使用熟悉的编辑器、调试器、测试框架和包管理器，无需学习专有的存储过程语言，也没有厂商锁定。

**亮点**

- 当前支持 Rust 与 AssemblyScript
- 使用熟悉工具链：Cargo、crate、语言服务器、调试器
- 可复用库与标准版本控制工作流
- 无需学习 PL/pgSQL 或 T-SQL

## 6. 内置 ORM 与类型安全

MuduDB 为查询结果映射到语言类型提供了一流支持。`Entity` trait 将 SQL 行转换为 Rust 结构体，`sql_stmt!` 和 `sql_params!` 宏在编译期捕获 SQL 与参数错误。`mgen` 工具还能直接根据 DDL 生成实体类型，在减少样板代码的同时避免引入笨重的外部 ORM。

**亮点**

- 通过 `Entity` trait 自动完成关系-对象映射
- `sql_stmt!` 与 `sql_params!` 提供编译期 SQL 校验
- `mgen` 根据 DDL 生成带类型的实体
- 在早期发现 schema 与查询不匹配

## 7. WebAssembly + MPK 打包

过程编译为 WebAssembly 组件，并打包成 `.mpk` 文件。MPK 是一个自包含的归档，包含 DDL、可选的初始数据、过程描述符以及一个或多个 WASM 模块。安装应用只需一条 `mcli app-install` 命令，使部署可复现且可版本化。

**亮点**

- 将过程编译为可移植的 WebAssembly 组件
- 把 schema、数据和逻辑打包进单个 `.mpk` 文件
- 一条 CLI 命令即可完成安装或升级
- 版本化部署，schema 与代码保持同步

## 8. 数据 proximity 处理

由于 MuduDB 过程运行在数据库内核内部，数据转换发生在数据旁边，而不是跨网络进行。一个过程可以在单一事务边界内查询多行、计算结果并更新状态，无需反复将中间数据序列化到远程客户端。

**亮点**

- 业务逻辑贴近数据执行
- 多步操作减少网络往返
- 复杂工作流的事务语义更简单
- 数据密集型任务吞吐更高

## 一览

| 维度 | 传统方案 | MuduDB |
|---|---|---|
| 业务逻辑运行位置 | 应用服务器 | 数据库内核内部 |
| 存储过程语言 | PL/pgSQL、T-SQL 等 | Rust、AssemblyScript |
| 开发与生产 | 不同代码或模式 | 同一套 Mudu Procedure 代码 |
| AI 生成代码验证 | 主要在运行期 | 编译期类型检查 |
| I/O 与并发模型 | 线程池 + 阻塞 I/O | `io_uring` + 每核 worker |
| 部署产物 | SQL 迁移 + 应用二进制 | 单个 `.mpk` 包 |
